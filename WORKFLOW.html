Outdoor Elements — AI Construction Takeoff

Current workflow & architecture · POC · updated 2026-06-17

Goal. A human estimator reads a construction drawing set and produces a takeoff — the measured quantity of every material — then checks it against the real human takeoff (the QTO, "quantity take-off"). The POC automates that so the AI does the same job in a fraction of the time. The project is built in stages, each a self-contained step.

Stage 1 — Upload & Page Selection DONE · verified

A web app. Upload the full drawing PDF; a worker keeps only the pages that hold measurable quantities and drops covers, notes, grading, lighting, details, sections and elevations — the same triage a human does first. Fully deterministic — no AI (the input is a true vector CAD export).

The selection rule (two filters)

FilterWhat it checksReads from the vector layer
1 · Sheet title Keep MATERIAL / HARDSCAPE / PLANTING / LANDSCAPE / REFERENCE PLAN. Drop NOTES, GRADING, LIGHTING, IRRIGATION, DETAIL, SECTION, ELEVATION, OVERALL. Title text read from the title-block corner only (not the whole page — avoids "SEE DETAIL" callouts).
2 · Legend richness Keep only pages with a real material legend. ≥ 3 distinct non-black/white vector fill colors (swatch complexity).

KEEP = Filter 1 passes AND Filter 2 passes.

Validated against ground truth

On 2811 Kirby - LANDSCAPE.pdf (35 pages) the rule reproduces the human's exact kept set — 6 / 6, zero false positives: L1.01, L1.02, L1.04, L5.01, L5.02, L5.03. The other 29 pages are dropped with a stated reason. 3 Playwright tests pass (upload selects 6 pages · click-to-preview opens · non-PDF rejected).

Scale math

PDF coordinates are in points (1 pt = 1/72 inch). With a sheet scale of 1 inch = S feet:

feet  = points     × (S / 72)
sqft  = sq_points  × (S / 72)²        e.g. L1.01: S = 10  →  sqft = pt² × (10/72)²

The "trust geometry, LLM only labels" principle: exact vector areas come from the PDF; Gemini is used only to attribute colors that collide or are shared between two materials.

Key finding — raw vs colored drawings

The measurement works on the COLORED sheet, not the raw drawing. The material areas are colored only in the human's QTO (≈ 21.9% of the page is filled). On the raw input drawing the same areas are not filled — only ~0.2% gray hatching — because the human creates each colored space by interpreting the plan. So the pipeline reaches 0% by measuring the colored QTO; measuring directly from the raw uncolored input is a separate, unsolved problem.

Every quick approach to detect spaces from the raw input was tested and failed to reach QTO accuracy:

ApproachOutcome
Pure vector closed polygonsNoisy — a material is hundreds of tiny tile/hatch elements, no single polygon = a space
Gemini outlines whole pageBoxes the legend column, not the plan areas
Colored line extractionColors every closed shape — unreadable
Gemini estimates areas from dimensions77–181% error + misreads the legend

Realistic path forward: human-in-the-loop click-to-measure (what commercial takeoff tools — Bluebeam, PlanSwift, Trimble — actually do). Full auto-detection is open R&D.

How to run

Stage 1 web app

# backend (from outdoor-elements/)
.venv/Scripts/python.exe -m uvicorn backend.main:app --port 8000
# frontend (from outdoor-elements/frontend/)
npm run dev        # http://localhost:5173
# tests
npx playwright test

Runs without Redis (eager mode). For real workers: start Redis, then celery -A backend.celery_app worker --pool=solo.

Measurement pipeline

# set the PDF, then run a sheet (0-based page index)
OE_PDF_PATH="…/2811 KIRBY QTO.pdf" python run_takeoff.py --page 0   # L1.01
python run_takeoff.py --all                                          # every sq-ft sheet
streamlit run app.py                                                 # review the results

GEMINI_API_KEY is read from .env — never hardcoded.
